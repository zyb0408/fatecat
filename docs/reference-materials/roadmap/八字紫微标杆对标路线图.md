# 八字与紫微标杆对标路线图

> 目标：在不继续堆新体系的前提下，优先把当前两个主要体系 `bazi` 与 `ziwei` 打磨到接近市面强排盘系统的完整度、准确性、可解释性和可审计性。
>
> 口径：本文是后续研发路线真相源，不声明未实现能力已经生产可用。所有新增能力必须继续遵循统一 capability 协议、evidence 规则、隐私治理和默认报告隔离。

## 1. 当前结论

| 体系 | 当前定位 | 当前成熟度 | 主要优势 | 主要短板 |
|---|---|---:|---|---|
| 综合八字 `bazi` | 默认 production 报告 | 60-70 | 真太阳时、节气、四柱、十神、五行、格局、用神、大运流年、神煞、称骨、evidence、命例 golden 已有生产骨架 | 命师级工具层不足，命例库不足，流日流时、人元司令、胎元胎息、定盘、合盘、专题断语和古籍规则库仍不完整 |
| 紫微斗数 `ziwei` | standalone production 报告 | 45-55 | 已复用 iztro，保留十二宫、命身宫、三方四正、四化落宫、运限四化、解释层和 evidence | 星曜百科、庙旺陷解释、格局识别、飞星派配置、借宫/对宫/夹宫规则、宫位专题和命例 golden 仍不足 |

核心判断：

- 八字和紫微都已经具备服务化生产骨架，但还不是顶级命师工作台。
- 下一阶段不新增六爻、奇门、大六壬等新体系；先把 `bazi`、`ziwei` 做深。
- 对标重点不是堆报告字数，而是排盘准确、字段完整、解释可追溯、命例可回归、前端可操作。

## 5. 执行路线

### Phase 1：八字准确性硬化

目标：先把“排盘和边界”锁死。

任务：

- BZ-P0-01：补早晚子时 golden。
- BZ-P0-02：补人元司令字段与 golden。
- BZ-P0-03：补起运算法多口径对照：顺逆、节气锚点、岁月日显示。
- BZ-P0-04：补强弱评分 golden：月令、通根、透干、藏干、五行分数。
- BZ-P0-05：补流日/流时结构化输出。

验收：

- `pytest` 增加八字边界命例。
- `scripts/acceptance.sh` 覆盖新增 golden。
- 每个关键结论必须能追到 `analysisEvidence.items.*.ruleIds`。

### Phase 2：八字规则库与解释层

目标：从“字段完整”升级为“命师级解释可追溯”。

任务：

- BZ-P0-06：建立格局规则 registry：正格、变格、从格、专旺、化气。
- BZ-P0-07：拆分用神口径：调候、扶抑、通关、病药。
- BZ-P0-08：建立十神组合解释库。
- BZ-P0-09：建立干支关系优先级：合化、三会、三合、冲刑害破、入库、拱合。
- BZ-P1-01：建立专题报告：事业、财运、婚姻、健康、学业、迁移。

验收：

- 不允许神煞、称骨影响格局和喜忌。
- Markdown 仍保持默认综合八字，不混入紫微或其他体系。
- JSON 可见 evidence，Markdown 不输出技术区。

### Phase 3：紫微完整盘面

目标：把 iztro 能力吃透，不在 FateCat 内重造底层斗数轮子。

任务：

- ZW-P0-01：补星曜分类：主星、辅星、杂曜、煞曜、吉曜。
- ZW-P0-02：补庙旺利陷字段与解释。
- ZW-P0-03：补宫位关系：对宫、夹宫、借宫、三方四正。
- ZW-P0-04：补四化飞入、冲照、会照字段。
- ZW-P0-05：补紫微 golden fixture：命盘、四化、宫位、运限。

验收：

- `ziweiChart` 保持 iztro 原始结构完整。
- `ziweiInterpretation` 只做解释编排，不覆盖底层星盘事实。
- 每个解释分组都有 rule ID。

### Phase 4：紫微解释库

目标：从“可输出盘”升级为“能解释盘”。

任务：

- ZW-P1-01：建立星曜百科索引。
- ZW-P1-02：建立主星组合规则库。
- ZW-P1-03：建立十二宫专题规则库。
- ZW-P1-04：建立格局识别：紫府同宫、机月同梁、杀破狼、府相朝垣、日月反背等。
- ZW-P1-05：建立本命、大限、流年三层联动解释。

验收：

- 输出“结论、依据、宫位、星曜、四化、限运、风险边界”。
- 不输出恐吓式断语。
- 不把紫微塞回综合八字默认报告。

### Phase 5：Web 专业化

目标：从“报告复制页”升级为“八字/紫微工作台”。

任务：

- UX-P1-01：八字工作台：四柱、十神、藏干、五行、格局、用神、大运流年可点击。
- UX-P1-02：紫微工作台：十二宫、星曜、四化、运限可点击。
- UX-P1-03：Markdown 与结构化 JSON 同源，不允许前端自行拼命理结论。
- UX-P1-04：示例数据继续固定北京/测试用户，真实地区不在前端示例泄露。

验收：

- Playwright 或 TestClient 覆盖页面结构。
- 复制 Markdown 与 API 输出一致。
- 所有非北京真实地区样例扫描为 0。

## 6. 工程约束

- 胶水原则优先：优先复用 `lunar-python`、`bazi-1`、`iztro`、现有 vendor 和公开成熟库。
- 自研只做连接、编排、规则索引、evidence 映射、报告组织和 UI。
- 八字是唯一默认 Markdown 生产报告；紫微必须 standalone。
- 黄历、梅花、六爻、奇门、大六壬等能力不得混入本路线的默认报告。
- 所有新增规则必须写入 `contracts/fate/classics_rule_index.json` 或后续分层规则索引。
- 所有新增核心断语必须有 golden 或快照测试。
- 所有外部资料必须记录来源、授权状态和是否可进入仓库。

## 7. 下一步建议

立即进入任务化：

1. 建立 `0002-bazi-ziwei-benchmark-hardening` 任务容器。
2. 先执行 BZ-P0-01 到 BZ-P0-05。
3. 同步执行 ZW-P0-01 到 ZW-P0-05。
4. 每完成一批就跑 `bash scripts/acceptance.sh`。
5. 通过后再进入 Web 工作台，不提前做大前端。
