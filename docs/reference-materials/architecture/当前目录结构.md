# 当前目录结构

## 目录重构结论

本仓库已经完成从旧式 `libs/ + docs/ + deploy/` 结构向 `assets/ + runtime/ + modules/` 结构的收敛。

核心原因：

1. `libs/` 把静态资产、外部仓库、数据库实库混在一起，边界模糊。
2. 顶层 `docs/`、`deploy/` 与 `assets/` 并列，职责分散。
3. 路径仍残留外部项目影子，导致仓库自洽性差。

## 当前推荐结构

```text
fatecat/
├── assets/
│   ├── config/        # 配置模板与配置文件
│   ├── data/          # 静态数据
│   ├── database/      # schema
│   ├── deploy/        # 打包与部署脚本
│   ├── docs/          # 文档资产索引、架构、运维、参考、路线图与归档
│   ├── fate/          # 输出字段 profile
│   └── vendor/        # 外部成熟仓库
├── runtime/
│   └── database/      # SQLite 实库
├── modules/
│   ├── fate_core/     # 纯命理分析内核
│   └── telegram/      # Bot / API 交付层
├── scripts/
└── tests/
```

## 合并映射关系

| 旧目录 | 新目录 | 说明 |
|--------|--------|------|
| `docs/` | `assets/docs/` | 文档归档到资产层 |
| `deploy/` | `assets/deploy/` | 部署脚本归档到资产层 |
| `libs/data/` | `domains/fate-analysis/data-products/` | 静态数据归档到资产层 |
| `libs/database/schema` | `assets/database/` | schema 归档到资产层 |
| `libs/database/*.db` | `runtime/database/` | 实库归入运行态 |
| `libs/external/` | `tools/reference-repos/` | 外部依赖归档到资产层 |

## 文档资产结构

```text
assets/docs/
├── README.md               # 文档资产索引
├── AGENTS.md               # 文档区维护边界
├── fatecat-readme-banner.svg
├── architecture/           # 架构图、目录结构、序列图
├── operations/             # 部署、启动、集成说明
├── reference/              # 功能清单、功能状态等当前参考
├── roadmap/                # 后续路线图与性能 TODO
├── vendor/                 # 外部成熟方案与供应链研究
├── prompts/                # 可复用 Prompt 资产
├── 生产故障/               # 事故与生产故障记录
├── 经验/                   # 实现经验与教训
└── archive/                # 本地历史输出、原始笔记等不作为当前真相源的材料
```

根层只保留索引、AGENTS 说明与 README banner；新的业务说明必须进入对应分区。

## 为什么这样更合理

- `assets/` 负责“不会因为运行而变化的东西”。
- `runtime/` 负责“程序一跑就会变化的东西”。
- `modules/` 负责“代码与业务能力”。

这样处理后：

1. 路径语义清晰。
2. 配置、数据、文档、部署脚本集中。
3. 纯分析模块与交付模块边界更稳。
4. 后续继续拆 `fate_core` 不会再被旧目录结构牵制。
