# 独立预测体系接入路线

> 日期：2026-05-08
> 范围：六爻、梅花、奇门、大六壬、风水九星、姓名合婚等 planned capability 的生产化顺序与复用边界。

## 总原则

- 每个体系必须先走 `assets/fate/capabilities/registry.json` 协议，再实现 `fate_core.usecases`。
- 成熟开源库优先；项目自研只做路径适配、输入输出归一、证据包装和风险边界。
- `status=production` 必须同时满足：provider/usecase、CLI/API 测试、evidence、隐私脱敏、文档和 acceptance。
- 未完成前保持 `planned`，不得进入默认综合八字 Markdown。

## 外部评测与资料

- `MingLi-Bench` 只作为 LLM 解释质量 benchmark，不作为生产算法。
- `bazi-ziwei-mingli-skill-v8.5.zip` 和 `classics/raw/` 只作为来源归档，需先按 `assets/data/classics/copyright_review.tsv` 完成分级。
- 通过复核的资料只能提炼为短规则、规则 ID、测试 fixture 或检索索引，不复制大段原文。

## 生产化检查清单

1. registry/profile 增加 capability，初始 `planned`。
2. provider/usecase 只依赖成熟库适配层或已复核数据。
3. executor 接入，planned 调用仍拒绝。
4. CLI/API 单元测试覆盖成功、缺字段、隐私脱敏。
5. evidence 至少包含 source、basis、ruleIds、risk。
6. README、功能状态、协议文档同步。
7. `bash scripts/acceptance.sh --with-dev --delivery-target api` 通过。
