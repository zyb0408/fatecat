# Planning Summary
按两个主体系的成熟度缺口拆分：先做共同治理和标杆证据，再做八字准确性与规则库，再做紫微盘面完整性与解释库，最后做 Web 工作台、验收和出货。
- 编译节点总数: 37
- 叶子执行项: 30
- 执行波次数: 12
- 当前任务必须遵守 `SPEC -> PLAN -> BUILD -> TEST -> REVIEW -> SHIP`

# Lifecycle Gates
- SPEC：路线图、标杆来源、范围边界、当前缺口已写入文档和任务上下文。
- PLAN：递归任务树、依赖图、执行波次、验收清单全部落盘并通过校验。
- BUILD：按 execution wave 执行叶子节点，每批实现绑定 TP 编号。
- TEST：新增测试覆盖对应 TP 验收，不允许只改代码不补测试。
- REVIEW：自审准确性、规则 evidence、隐私、输出边界和导出包卫生。
- SHIP：提交、推送、CI 通过，任务 closeout 更新。
- 不得跳过未闭合 gate 直接进入下一阶段。

# Simplest Path
不新增新体系，不替换底层成熟库；先把现有 bazi/ziwei 的计算事实、规则 evidence、命例 golden 和输出体验补齐。

# Split Strategy
顶层按治理域拆分，叶子按可独立实现和验证的交付物拆分；P0 先覆盖准确性和 evidence，P1 再覆盖专题和工作台。
