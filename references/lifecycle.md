# FateCat Skill 生命周期模型

## 目标

把 FateCat 从“能跑起来”推进到“能持续演化、能交付、能运维、能体面退役”的连续管理对象。

## 阶段链条

### 1. 需求

- 产物：`00-context.md`、`01-requirements.md`
- 目标：把原始诉求压成明确边界、输入、输出、风险与验收
- 通过标准：需求不再停留在口头描述，可以驱动原型验证

### 2. 原型

- 产物：`02-prototype.md`
- 目标：用最小实现验证链路能跑通
- 通过标准：至少有一条可执行命令或一组可复现步骤证明功能可行

### 3. 迭代

- 产物：`03-iteration.md`
- 目标：把临时原型打磨成连续可交付版本
- 通过标准：迭代目标、已知缺口、回归风险和下一轮计划可追踪

### 4. 成熟方案重构

- 产物：`04-mature-refactor.md`
- 目标：识别自研临时件，把它替换成更稳的成熟组件、成熟模式或成熟元素
- 通过标准：替换动机、收益、兼容风险、回滚方案都写清楚

### 5. 生产优化

- 产物：`05-production-hardening.md`
- 目标：在性能、稳定性、可部署性、可配置性上达成生产最小面
- 通过标准：明确 SLO、瓶颈、回归检查与上线前门禁

### 6. 运维加固

- 产物：`06-operations.md` 与独立 ops bundle
- 目标：让 Agent 或运维人员在故障时有健康检查、恢复路径、证据快照和知识沉淀
- 通过标准：至少能一键采集健康快照，并给出明确重启与排障入口

### 7. 退役

- 产物：`07-retirement.md`
- 目标：在停用、迁移、冻结或并入新系统时，保留依赖、数据、替代路径与用户影响说明
- 通过标准：退役条件、数据处置、下游通知与归档边界清楚

## 推荐推进方式

- 先创建生命周期包，再做实现。
- 每推进一阶段，更新对应阶段文档，不把所有结论塞进一个大 README。
- 每次进入新阶段前，看 `stage-gates.md`，确认前一阶段的退出条件是否满足。
- 每次接近交付或发布前，运行健康检查并采集运维包。

## 与企业源码根的关系

- `domains/*/services/*` 是目标业务源码根。
- lifecycle 资产位于 `docs/reference-materials/lifecycle/`；运行时和源码不依赖 lifecycle packs。
- 生命周期包是治理真相源，用来记录为什么做、做到哪、还有什么风险。
- 两者不能混写：业务实现优先放 `domains/*/services/*`，阶段证据放 `governance/` 或受控 lifecycle 资产区。
